## 1. Architectural Philosophy

KPNHCIS Backend is a **Headless HCIS** — the backend exposes a pure REST API with no knowledge of the frontend UI. It is built on three pillars:

| Pillar | Technology | Role |
|---|---|---|
| **Transport** | NestJS + TypeScript | HTTP routing, validation, authentication |
| **Authorization** | CASL + Redis | Attribute-based access control, cached per user |
| **Persistence** | Prisma ORM + PostgreSQL 16 | Relational schema + JSONB hybrid columns |

**Core rules that are never negotiated:**

- No business logic in Controllers. Controllers translate HTTP → Service calls.
- No raw SQL in Services unless a JSONB GIN query is explicitly required.
- No hard deletes. Always use `deletedAt` soft-delete.
- No bypassing the CASL + JWT guard stack on any authenticated route.
- JSONB payload columns are always versioned (`_v: 1`). Schema changes increment `_v`.

## 2. Layer Responsibilities

```
HTTP Request
     │
     ▼
┌─────────────────────────────────┐
│         Controller Layer        │  Route definition, DTO validation,
│   (leave-request.controller.ts) │  guard decoration, @CurrentUser()
│                                 │  extraction. NO logic beyond this.
└────────────────┬────────────────┘
                 │ calls
                 ▼
┌─────────────────────────────────┐
│          Service Layer          │  Business rules, XState machine
│   (leave-request.service.ts)    │  interaction, Prisma calls,
│                                 │  transaction management, logging.
└────────────────┬────────────────┘
                 │ calls
                 ▼
┌─────────────────────────────────┐
│      Prisma / Repository        │  PrismaService (extends PrismaClient).
│   (src/prisma/prisma.service.ts)│  All SQL is generated by Prisma unless
│                                 │  a JSONB GIN query demands $queryRaw.
└─────────────────────────────────┘
```

### Controller Layer

**Allowed:**
- Declaring route methods (`@Get`, `@Post`, `@Patch`, `@Delete`).
- Applying guards (`@UseGuards(JwtAuthGuard, PermissionsGuard)`).
- Declaring permissions (`@CheckPermissions(['action', 'Subject'])`).
- Extracting path params, query params, body DTOs, and the current user.
- Calling exactly **one** service method per handler.

**Forbidden:**
- Any `if/else` business logic.
- Direct `PrismaService` calls (except read-only list queries that belong to no other service).
- `try/catch` blocks — exception filters handle errors globally.

### Service Layer

**Allowed:**
- All business rules, validations that require DB state, and state transitions.
- Injecting `PrismaService` directly.
- Using `prisma.$transaction()` for multi-table atomic operations.
- Logging via `private readonly logger = new Logger(ServiceName.name)`.
- Throwing NestJS HTTP exceptions (`NotFoundException`, `ConflictException`, etc.).

**Forbidden:**
- HTTP-specific concerns (status codes, headers, response shaping).
- Calling another module's controller.

### Prisma / Repository Layer

`PrismaService` is a thin singleton that wraps `PrismaClient`. It is registered globally via `PrismaModule`. All other modules receive it automatically without importing `PrismaModule` explicitly.

## 4. Hybrid JSONB Architecture

Every domain model that requires **flexible metadata** has a `payload Json? @db.JsonB` column alongside strictly typed relational columns. The rule is:

```
Core identity fields   → Relational columns (typed, indexed, queryable by Prisma)
Flexible metadata      → payload JSONB column (versioned, GIN-indexed)
```

**payload always includes a schema version sentinel `_v`.**

```typescript
// The internal TypeScript interface for any payload column.
// Define this inside the service file that owns the model — do not export it.
interface RewardSanctionPayload {
  _v: number;                   // increment when shape changes; currently 1
  attachments?: Attachment[];   // evidence documents, award certificates
  decisionReference?: string;   // SK / surat keputusan reference number
  points?: number;              // disciplinary/reward point value for scoring
  [key: string]: unknown;       // allows future dynamic fields without migration
}

interface Attachment {
  fileKey:    string;   // object storage key (S3 / local)
  fileName:   string;
  mimeType:   string;
  uploadedAt: string;   // ISO 8601
}
```

**When to add a field to payload vs. a column** → See [Section 6.1](#61-migration-vs-jsonb--decision-table).

## 5. Step-by-Step: Building the `RewardSanction` Module

This section walks through the creation of a complete, production-ready `RewardSanction` module. Reward/sanction records track disciplinary actions and performance awards for employees.

---

### 5.1 Prisma Schema

Open `prisma/schema.prisma` and add the following. Keep all enums at the top of the enum section and all models in the model section.

```prisma
// Add to the ENUMS section ─────────────────────────────────────────────────

enum RewardSanctionType {
  REWARD
  SANCTION
}

// Add to the MODELS section ────────────────────────────────────────────────

// =============================================================================
// REWARD / SANCTION — HR disciplinary actions and performance awards.
// Soft-deleted via deletedAt; never hard-deleted.
// The `payload` JSONB column stores attachments, SK references, and point values.
// =============================================================================

model RewardSanction {
  id             String             @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId         String             @map("user_id") @db.Uuid
  user           User               @relation("ReceivedRewardSanctions", fields: [userId], references: [id])
  issuedByUserId String             @map("issued_by_user_id") @db.Uuid
  issuedBy       User               @relation("IssuedRewardSanctions", fields: [issuedByUserId], references: [id])

  type           RewardSanctionType
  category       String             // e.g. "KINERJA", "DISIPLIN", "INOVASI"
  title          String
  description    String
  effectiveDate  DateTime           @map("effective_date") @db.Date
  expiryDate     DateTime?          @map("expiry_date") @db.Date
  isActive       Boolean            @default(true) @map("is_active")

  // Hybrid JSONB: attachments, SK references, scoring metadata
  payload        Json?              @db.JsonB

  createdAt      DateTime           @default(now()) @map("created_at")
  updatedAt      DateTime           @updatedAt @map("updated_at")
  deletedAt      DateTime?          @map("deleted_at")  // soft delete — never hard delete

  @@index([userId])
  @@index([issuedByUserId])
  @@index([type])
  @@index([isActive])
  @@map("reward_sanctions")
}
```

> **Warning**: The `User` model now needs two named relations. Add these two lines inside the `User` model:
>
> ```prisma
> model User {
>   // ... existing fields ...
>   receivedRewardSanctions RewardSanction[] @relation("ReceivedRewardSanctions")
>   issuedRewardSanctions   RewardSanction[] @relation("IssuedRewardSanctions")
> }
> ```

### 5.2 Running the Migration

After editing the schema, generate and apply the migration:

```bash
# Generate SQL and apply to the local database in one step.
# Use a descriptive migration name — it becomes a permanent filename.
npx prisma migrate dev --name add_reward_sanction_module

# Regenerate the Prisma Client so TypeScript picks up the new types.
# migrate dev does this automatically, but run it manually if you only ran migrate deploy.
npx prisma generate
```

> **Rule**: Never use `prisma db push` in any environment that has a migration history. `db push` bypasses the migration log and will cause `prisma migrate status` to show drift in staging/production.

### 5.4 Data Transfer Objects (DTOs)

DTOs live in `src/reward-sanction/dto/`. They use `class-validator` decorators for automatic validation — NestJS's global `ValidationPipe` in `main.ts` enforces them.

**`create-reward-sanction.dto.ts`**

```typescript
// src/reward-sanction/dto/create-reward-sanction.dto.ts

import {
  IsDateString,
  IsEnum,
  IsNotEmpty,
  IsObject,
  IsOptional,
  IsString,
  IsUUID,
} from 'class-validator';
import { RewardSanctionType } from '@prisma/client';

export class CreateRewardSanctionDto {
  @IsUUID('4')
  userId: string;

  @IsEnum(RewardSanctionType)
  type: RewardSanctionType;

  @IsString()
  @IsNotEmpty()
  category: string;

  @IsString()
  @IsNotEmpty()
  title: string;

  @IsString()
  @IsNotEmpty()
  description: string;

  @IsDateString()
  effectiveDate: string;

  @IsDateString()
  @IsOptional()
  expiryDate?: string;

  /**
   * Arbitrary metadata: attachments, SK references, points, etc.
   * The service layer merges this into the versioned payload structure.
   * Clients do NOT need to supply _v — the service always sets it.
   */
  @IsObject()
  @IsOptional()
  payload?: Record<string, unknown>;
}
```

**`update-reward-sanction.dto.ts`**

```typescript

import { Type } from 'class-transformer';
import { RewardSanctionType } from '@prisma/client';

export class RewardSanctionQueryDto {

  userId?: string;

  @IsOptional()
  type?: RewardSanctionType;

  @IsOptional()
  category?: string;

  /**
   * When true, only returns non-deleted records.
   * Defaults to true (soft-deleted records are excluded by default).

  @IsInt()
  @Min(0)
  @IsOptional()

  @IsInt()
  @Min(1)
  @Max(100)
  @IsOptional()
  take?: number = 20;
}
```

### 5.5 Service Layer

The service is the core of every module. It owns all business logic and is the only layer permitted to call `PrismaService`.

```typescript
// src/reward-sanction/reward-sanction.service.ts

import {
  BadRequestException,
  Injectable,
  Logger,
  NotFoundException,
} from '@nestjs/common';
import { Prisma, RewardSanction, RewardSanctionType } from '@prisma/client';

import { PrismaService } from '../prisma/prisma.service';
import { CreateRewardSanctionDto } from './dto/create-reward-sanction.dto';
import { RewardSanctionQueryDto } from './dto/reward-sanction-query.dto';
import { UpdateRewardSanctionDto } from './dto/update-reward-sanction.dto';

// =============================================================================
// Internal JSONB payload interface — versioned for future migrations.
// NOT exported: payload shape is an implementation detail of this service.
// =============================================================================

interface Attachment {
  fileKey:    string;
  fileName:   string;
  mimeType:   string;
  uploadedAt: string; // ISO 8601
}

interface RewardSanctionPayload {
  _v: number;                  // always 1 for newly created records
  attachments?: Attachment[];  // evidence documents, award certificates
  decisionReference?: string;  // SK / surat keputusan reference number
  points?: number;             // HR scoring point value
  [key: string]: unknown;      // allows future keys without breaking existing records
}

// =============================================================================
// RawRewardSanctionRow — shape returned by $queryRaw JSONB queries.
//
// $queryRaw returns PostgreSQL's native column names (snake_case) unless the
// SQL aliases them. All $queryRaw calls in this service alias to camelCase so
// this interface maps directly to JS objects with no secondary transformation.
// =============================================================================

export interface RawRewardSanctionRow {
  id:             string;
  userId:         string;
  issuedByUserId: string;
  type:           RewardSanctionType;
  category:       string;
  title:          string;
  description:    string;
  effectiveDate:  Date;
  expiryDate:     Date | null;
  isActive:       boolean;
  payload:        Record<string, unknown> | null;
  createdAt:      Date;
  updatedAt:      Date;
  deletedAt:      Date | null;
}

@Injectable()
export class RewardSanctionService {
  private readonly logger = new Logger(RewardSanctionService.name);

  constructor(private readonly prisma: PrismaService) {}

  // ---------------------------------------------------------------------------
  // create — atomic: inserts the record and writes an initial audit entry
  //
  // prisma.$transaction() ensures that if the audit log insert fails, the main
  // record insert is also rolled back. Both rows land in the same DB transaction
  // or neither does. This is the standard pattern for any "create + audit" flow.
  // ---------------------------------------------------------------------------
  async create(
    dto: CreateRewardSanctionDto,
    issuedByUserId: string,
  ): Promise<RewardSanction> {
    // Validate that the target employee exists before writing anything.
    const targetUser = await this.prisma.user.findUnique({
      where:  { id: dto.userId },
      select: { id: true, fullName: true },
    });

    if (!targetUser) {
      throw new NotFoundException(`User ${dto.userId} not found`);
    }

    // Build the versioned payload. The client-supplied payload (extra metadata)
    // is merged in AFTER the versioned sentinel fields to prevent the client
    // from overwriting _v with a different value.
    const initialPayload: RewardSanctionPayload = {
      _v:          1,
      attachments: [],
      ...(dto.payload ?? {}),  // client-supplied metadata is merged last
    };

    // ---------------------------------------------------------------------------
    // Atomic transaction: create the main record + write the first audit row.
    //
    // $transaction receives an async callback that receives `tx`, a scoped
    // Prisma client bound to the current transaction. All Prisma calls inside
    // the callback use `tx`, NOT `this.prisma`. If any call throws, Postgres
    // automatically rolls back both operations.
    // ---------------------------------------------------------------------------
    const created = await this.prisma.$transaction(
      async (tx: Prisma.TransactionClient) => {
        const record = await tx.rewardSanction.create({
          data: {
            userId:         dto.userId,
            issuedByUserId,
            type:           dto.type,
            category:       dto.category,
            title:          dto.title,
            description:    dto.description,
            effectiveDate:  new Date(dto.effectiveDate),
            expiryDate:     dto.expiryDate ? new Date(dto.expiryDate) : null,
            isActive:       true,
            payload:        initialPayload as unknown as Prisma.InputJsonValue,
          },
          include: {
            user:      { select: { id: true, fullName: true, employeeId: true } },
            issuedBy:  { select: { id: true, fullName: true } },
          },
        });

        // Example of a second write inside the same transaction.
        // Here we append an event to a hypothetical audit log table.
        // If your module has no second write, omit this block entirely.
        // 
        // await tx.auditLog.create({
        //   data: {
        //     entityType: 'RewardSanction',
        //     entityId:   record.id,
        //     action:     'CREATE',
        //     actorId:    issuedByUserId,
        //     snapshot:   record as unknown as Prisma.InputJsonValue,
        //   },
        // });

        return record;
      },
    );

    this.logger.log(
      `RewardSanction created: ${created.id} ` +
      `type=${created.type} user=${dto.userId} issuedBy=${issuedByUserId}`,
    );

    return created;
  }

  // ---------------------------------------------------------------------------
  // findAll — paginated list with optional filters.
  //
  // Soft-deleted records are excluded by default (deletedAt: null).
  // HR admins can pass excludeDeleted=false to see the full audit trail.
  // ---------------------------------------------------------------------------
  async findAll(
    query: RewardSanctionQueryDto,
  ): Promise<{ data: RewardSanction[]; total: number }> {
    const where: Prisma.RewardSanctionWhereInput = {
      ...(query.userId   && { userId:   query.userId }),
      ...(query.type     && { type:     query.type }),
      ...(query.category && { category: query.category }),
      // Apply soft-delete filter unless the caller explicitly wants deleted records.
      ...(query.excludeDeleted !== false && { deletedAt: null }),
    };

    const [data, total] = await this.prisma.$transaction([
      this.prisma.rewardSanction.findMany({
        where,
        skip:    query.skip ?? 0,
        take:    query.take ?? 20,
        orderBy: { createdAt: 'desc' },
        include: {
          user:     { select: { id: true, fullName: true, employeeId: true } },
          issuedBy: { select: { id: true, fullName: true } },
        },
      }),
      this.prisma.rewardSanction.count({ where }),
    ]);

    return { data, total };
  }

  // ---------------------------------------------------------------------------
  // findOne — single record. Throws 404 if not found OR already soft-deleted.
  // ---------------------------------------------------------------------------
  async findOne(id: string): Promise<RewardSanction> {
    const record = await this.prisma.rewardSanction.findFirst({
      where: {
        id,
        deletedAt: null, // exclude soft-deleted records from normal lookups
      },
      include: {
        user:     { select: { id: true, fullName: true, employeeId: true } },
        issuedBy: { select: { id: true, fullName: true } },
      },
    });

    if (!record) {
      throw new NotFoundException(`RewardSanction ${id} not found`);
    }

    return record;
  }

  // ---------------------------------------------------------------------------
  // update — partial update with safe JSONB merge.
  //
  // The JSONB payload is merged at the top level, not replaced wholesale.
  // This prevents a partial update call from wiping fields the caller
  // did not include in the request body.
  //
  // Example: existing payload = { _v: 1, attachments: [...], points: 5 }
  //          dto.payload       = { decisionReference: "SK-001-2026" }
  //          merged result     = { _v: 1, attachments: [...], points: 5, decisionReference: "SK-001-2026" }
  // ---------------------------------------------------------------------------
  async update(
    id: string,
    dto: UpdateRewardSanctionDto,
  ): Promise<RewardSanction> {
    const existing = await this.findOne(id); // throws 404 if not found or deleted

    // ---------------------------------------------------------------------------
    // Safe JSONB merge: spread the existing payload first, then overlay the
    // incoming dto.payload. This is a shallow merge — nested arrays such as
    // `attachments` must be sent in full if the caller intends to modify them.
    //
    // NEVER do: payload: dto.payload  — this replaces the entire object
    // ALWAYS do the spread merge below.
    // ---------------------------------------------------------------------------
    const existingPayload = (existing.payload ?? {}) as RewardSanctionPayload;
    const mergedPayload: RewardSanctionPayload = {
      ...existingPayload,
      ...(dto.payload ?? {}),
      _v: existingPayload._v ?? 1, // preserve version; clients cannot downgrade it
    };

    const { payload: _ignored, userId: _userId, ...scalarUpdates } = dto;

    return this.prisma.rewardSanction.update({
      where: { id },
      data: {
        ...scalarUpdates,
        ...(dto.effectiveDate && { effectiveDate: new Date(dto.effectiveDate) }),
        ...(dto.expiryDate    && { expiryDate:    new Date(dto.expiryDate)    }),
        payload: mergedPayload as unknown as Prisma.InputJsonValue,
      },
      include: {
        user:     { select: { id: true, fullName: true, employeeId: true } },
        issuedBy: { select: { id: true, fullName: true } },
      },
    });
  }

  // ---------------------------------------------------------------------------
  // addAttachment — appends a single attachment to payload.attachments[].
  //
  // This method demonstrates how to safely mutate a deeply nested array inside
  // a JSONB column without replacing unrelated fields.
  //
  // Pattern:
  //   1. Read the current record (findOne throws 404 if missing)
  //   2. Cast payload to the typed interface
  //   3. Push the new element onto the nested array
  //   4. Write the entire payload back (Prisma does not support atomic JSON array push)
  // ---------------------------------------------------------------------------
  async addAttachment(
    id: string,
    attachment: Attachment,
  ): Promise<RewardSanction> {
    const existing = await this.findOne(id);

    // Step 1: Cast JSONB to the typed interface.
    // `existing.payload` is typed as `Prisma.JsonValue | null` by the generated client.
    // Cast it explicitly — TypeScript cannot infer the shape of JSONB columns.
    const existingPayload = (existing.payload ?? { _v: 1, attachments: [] }) as RewardSanctionPayload;

    // Step 2: Defensively ensure the array exists (older records may predate this field).
    const existingAttachments: Attachment[] = Array.isArray(existingPayload.attachments)
      ? existingPayload.attachments
      : [];

    // Step 3: Build the updated payload with the new attachment appended.
    const updatedPayload: RewardSanctionPayload = {
      ...existingPayload,
      attachments: [...existingAttachments, attachment],
    };

    // Step 4: Persist the full updated payload back to Postgres.
    return this.prisma.rewardSanction.update({
      where: { id },
      data:  { payload: updatedPayload as unknown as Prisma.InputJsonValue },
    });
  }

  // ---------------------------------------------------------------------------
  // softDelete — sets deletedAt to now. The record remains in the DB for audits.
  //
  // RULE: This is the ONLY way to remove a record. Hard deletes are forbidden.
  // ---------------------------------------------------------------------------
  async softDelete(id: string, deletedByUserId: string): Promise<RewardSanction> {
    await this.findOne(id); // throws 404 if already deleted or not found

    const deleted = await this.prisma.rewardSanction.update({
      where: { id },
      data:  { deletedAt: new Date() },
    });

    this.logger.log(
      `RewardSanction soft-deleted: ${id} by user ${deletedByUserId}`,
    );

    return deleted;
  }

  // ---------------------------------------------------------------------------
  // findByPayloadContainment — GIN-accelerated JSONB @> containment query.
  //
  // Uses the GIN index on the payload column. This is the only method in this
  // service that uses $queryRaw. Use it when:
  //   - You need to filter on a key nested inside the JSONB payload
  //   - Prisma's `path` filter would generate a sequential scan
  //
  // Example usage:
  //   findByPayloadContainment({ points: 10 })
  //   findByPayloadContainment({ decisionReference: "SK-001-2026" })
  //
  // DO NOT use `WHERE payload->>'points' = '10'` — that hits no GIN index.
  // USE    `WHERE payload @> '{"points": 10}'::jsonb` — this hits the GIN index.
  // ---------------------------------------------------------------------------
  async findByPayloadContainment(
    filter: Record<string, unknown>,
    options?: { skip?: number; take?: number },
  ): Promise<{ data: RawRewardSanctionRow[]; total: number }> {
    const { skip = 0, take = 20 } = options ?? {};
    const jsonFilter = JSON.stringify(filter);

    const [data, countRows] = await Promise.all([
      this.prisma.$queryRaw<RawRewardSanctionRow[]>`
        SELECT
          id,
          user_id            AS "userId",
          issued_by_user_id  AS "issuedByUserId",
          type,
          category,
          title,
          description,
          effective_date     AS "effectiveDate",
          expiry_date        AS "expiryDate",
          is_active          AS "isActive",
          payload,
          created_at         AS "createdAt",
          updated_at         AS "updatedAt",
          deleted_at         AS "deletedAt"
        FROM reward_sanctions
        WHERE payload @> ${jsonFilter}::jsonb
          AND deleted_at IS NULL
        ORDER BY created_at DESC
        LIMIT ${take} OFFSET ${skip}
      `,
      this.prisma.$queryRaw<[{ count: bigint }]>`
        SELECT COUNT(*)::bigint AS count
        FROM reward_sanctions
        WHERE payload @> ${jsonFilter}::jsonb
          AND deleted_at IS NULL
      `,
    ]);

    return {
      data,
      total: Number(countRows[0].count),
    };
  }
}
```

```typescript
 *
 * ─── Auth model ───────────────────────────────────────────────────────────────
 *   JwtAuthGuard     — validates Bearer token; populates request.user
 *   PermissionsGuard — evaluates @CheckPermissions() against the user's CASL
 *                      ability (built from DB roles, cached in Redis)
 *
 * Class-level @CheckPermissions(['read', 'RewardSanction']) applies to every

  @Patch(':id')

  @Delete(':id')
```

## 6. Prisma Best Practices

### 6.1 Migration vs. JSONB — Decision Table

Before adding any new field, use this table to decide where it lives:

| Criterion | Relational Column (Migration) | JSONB payload |
|---|---|---|
| **Queried in `WHERE` by Prisma** | ✅ Always — B-tree index is efficient | ⚠️ Only via `$queryRaw` + GIN |
| **Used in `ORDER BY`** | ✅ Fast with an index | ❌ Not practical |
| **Joined to another table** | ✅ Required for FK constraints | ❌ Never — JSONB has no FK |
| **Shape is fixed and known at design time** | ✅ Preferred | — |
| **Shape varies per record (polymorphic)** | ❌ Would require nullable columns | ✅ Ideal for JSONB |
| **Requires DB-level `NOT NULL` constraint** | ✅ Required | ❌ JSONB has no column constraints |
| **Appended to from multiple code paths** | ❌ Migrations are costly | ✅ JSONB merge is flexible |
| **Audit trail / event log arrays** | ❌ Would need a join table | ✅ Append to `payload.transitions[]` |
| **User-defined custom fields (dynamic schema)** | ❌ Requires ALTER TABLE | ✅ JSONB is designed for this |

**Practical examples from this project:**

```
userId, status, effectiveDate          → Relational columns (filtered, sorted, joined)
approvalChain, currentStepIndex        → JSONB payload (XState context, not joined)
attachments[], decisionReference       → JSONB payload (polymorphic, per-record)
positionIds[]                          → JSONB payload (variable length, user-specific)
```

---

### 6.2 Safe JSONB Querying in TypeScript

**Rule 1: Always cast `Prisma.JsonValue` to your typed interface before reading nested fields.**

```typescript
// ❌ WRONG — TypeScript will error: "Property 'attachments' does not exist on type JsonValue"
const attachments = record.payload.attachments;

// ✅ CORRECT — cast first, then access safely
const typedPayload = (record.payload ?? {}) as RewardSanctionPayload;
const attachments = typedPayload.attachments ?? [];
```

**Rule 2: Always provide a fallback when reading nested JSONB arrays.**

```typescript
// Older records may not have the `attachments` array (added in v1.1 of the payload schema).
// Using Array.isArray() guards against `null`, `undefined`, and non-array values.
const existingAttachments: Attachment[] = Array.isArray(typedPayload.attachments)
  ? typedPayload.attachments
  : [];
```

**Rule 3: When writing back to JSONB, always cast through `unknown` first.**

```typescript
// Prisma's InputJsonValue type does not accept user-defined interfaces directly.
// The double-cast (as unknown as Prisma.InputJsonValue) is the standard pattern.

// ❌ WRONG — TypeScript error: interface is not assignable to InputJsonValue
await prisma.rewardSanction.update({
  data: { payload: updatedPayload },
});

// ✅ CORRECT
await prisma.rewardSanction.update({
  data: { payload: updatedPayload as unknown as Prisma.InputJsonValue },
});
```

**Rule 4: Use `@>` (containment) for GIN-indexed JSONB queries. Never use `->>` in `WHERE` clauses.**

```typescript
// ❌ WRONG — forces a sequential scan; ignores the GIN index entirely
this.prisma.$queryRaw`
  SELECT * FROM reward_sanctions
  WHERE payload->>'decisionReference' = 'SK-001-2026'
`;

// ✅ CORRECT — GIN index is used; O(log n) lookup
const filter = JSON.stringify({ decisionReference: 'SK-001-2026' });
this.prisma.$queryRaw`
  SELECT * FROM reward_sanctions
  WHERE payload @> ${filter}::jsonb
`;
```

**Rule 5: Always alias $queryRaw columns to camelCase.**

```typescript
// ❌ WRONG — returns snake_case keys; breaks interface mapping
this.prisma.$queryRaw`SELECT user_id, effective_date FROM reward_sanctions`;
// result[0].user_id ← TypeScript doesn't know this key

// ✅ CORRECT — alias every column; maps directly to your RawRow interface
this.prisma.$queryRaw<RawRewardSanctionRow[]>`
  SELECT
    user_id           AS "userId",
    effective_date    AS "effectiveDate"
  FROM reward_sanctions
`;
// result[0].userId ← fully typed
```

---

### 6.3 Soft Delete — Never Hard Delete

Every model that represents an HR record must have a soft-delete column:

```prisma
deletedAt DateTime? @map("deleted_at")
```

**Enforcement rules:**

1. **All standard read queries must filter `deletedAt: null`** unless the caller has an explicit audit/archive use case.

   ```typescript
   // ✅ Standard findOne — excludes soft-deleted records
   await this.prisma.rewardSanction.findFirst({
     where: { id, deletedAt: null },
   });
   ```

2. **Soft-delete by setting `deletedAt` to now, not by calling `delete()`.**

   ```typescript
   // ❌ NEVER
   await this.prisma.rewardSanction.delete({ where: { id } });

   // ✅ ALWAYS
   await this.prisma.rewardSanction.update({
     where: { id },
     data:  { deletedAt: new Date() },
   });
   ```

3. **Log who performed the deletion.** The service must receive `deletedByUserId` and emit a log entry or append to `payload.transitions`.

4. **`isActive` is not the same as soft-delete.** `isActive = false` deactivates a record but keeps it fully queryable. `deletedAt` hides it from all standard queries. Use `isActive` for business state changes; use `deletedAt` for data lifecycle management.

### 6.4 Atomic Transactions

Use `prisma.$transaction()` whenever two or more DB writes must succeed or fail together.

**Pattern A: Interactive transaction (callback form) — use this when write B depends on the result of write A.**

```typescript
const result = await this.prisma.$transaction(
  async (tx: Prisma.TransactionClient) => {
    // Write A: create the main record
    const record = await tx.rewardSanction.create({ data: { ... } });

    // Write B: uses record.id from Write A — cannot run before A completes
    await tx.someAuditTable.create({
      data: { entityId: record.id, action: 'CREATE' },
    });

    // The callback's return value becomes the return value of $transaction()
    return record;
  },
);
```

**Pattern B: Batch transaction (array form) — use this when writes are independent.**

```typescript
// Both queries run in a single transaction but neither depends on the other.
const [data, total] = await this.prisma.$transaction([
  this.prisma.rewardSanction.findMany({ where, skip, take }),
  this.prisma.rewardSanction.count({ where }),
]);
```

> **Warning**: Inside a `$transaction` callback, **always use `tx`**, not `this.prisma`. Using `this.prisma` inside the callback executes queries outside the transaction, defeating the atomicity guarantee. This is a common and silent bug.

```typescript
// ❌ SILENT BUG — the second write is outside the transaction
async (tx) => {
  await tx.rewardSanction.create({ ... });       // ← inside tx ✓
  await this.prisma.auditLog.create({ ... });     // ← OUTSIDE tx ✗ — not atomic!
}

// ✅ CORRECT — both writes use tx
async (tx) => {
  await tx.rewardSanction.create({ ... });        // ← inside tx ✓
  await tx.auditLog.create({ ... });              // ← inside tx ✓
}
```

## 7. CASL Permission Seed Reference

After creating a new module, insert the required permissions into the `permissions` table and assign them to the appropriate roles. This is typically done via a Prisma seed script (`prisma/seed.ts`) or through the admin API.

The standard permission matrix for a new CRUD module:

| action | subject | conditions | Assigned to role |
|---|---|---|---|
| `create` | `RewardSanction` | `null` | `HR_ADMIN`, `SUPERVISOR` |
| `read` | `RewardSanction` | `null` | `HR_ADMIN`, `JS`, `SUPERVISOR` |
| `read` | `RewardSanction` | `{ "userId": "${user.id}" }` | `EMPLOYEE` (own records only) |
| `update` | `RewardSanction` | `null` | `HR_ADMIN` |
| `delete` | `RewardSanction` | `null` | `HR_ADMIN` |

The `${user.id}` placeholder is resolved at runtime by `CaslAbilityFactory.interpolateConditions()`. An employee with the scoped `read` permission will only receive records where `userId` matches their own ID.

## 8. New Module Checklist

Use this checklist for every new module before submitting a pull request.

**Schema & Database**
- [ ] Prisma model added to `schema.prisma` with `@map` for all fields
- [ ] Enum added to the ENUMS section (if applicable)
- [ ] `deletedAt DateTime? @map("deleted_at")` present on all HR record models
- [ ] `@@index` declared for all foreign keys and commonly filtered columns
- [ ] Migration generated with a descriptive name (`npx prisma migrate dev --name <name>`)
- [ ] `prisma generate` run after migration

**CASL**
- [ ] `SubjectName` union updated in `src/casl/casl.types.ts`
- [ ] Permission rows inserted via seed or admin API
- [ ] Correct roles assigned to each permission

**Code Files**
- [ ] `create-<module>.dto.ts` — all fields decorated with `class-validator`
- [ ] `update-<module>.dto.ts` — extends `PartialType(Create...)` or declares only updatable fields
- [ ] `<module>-query.dto.ts` — pagination (`skip`, `take`) + relevant filter fields
- [ ] `<module>.service.ts` — `Logger` instantiated; all DB calls use `PrismaService`
- [ ] `<module>.controller.ts` — `@UseGuards(JwtAuthGuard, PermissionsGuard)` at class level
- [ ] Every route handler has `@CheckPermissions(...)` at method or class level
- [ ] No business logic in controller methods — single service call per handler
- [ ] `<module>.module.ts` — imports `[AuthModule, CaslModule]`
- [ ] Module registered in `src/app.module.ts`

**JSONB Payload**
- [ ] `payload` interface defined privately inside the service (not exported)
- [ ] `_v: 1` included in all `initialPayload` objects on creation
- [ ] All JSONB reads cast through the typed interface (`as RewardSanctionPayload`)
- [ ] All JSONB writes cast through `as unknown as Prisma.InputJsonValue`
- [ ] Update operations use spread merge (`{ ...existingPayload, ...dto.payload }`)
- [ ] `$queryRaw` used only for GIN `@>` queries; all other queries use Prisma API
- [ ] All `$queryRaw` SELECT columns aliased to camelCase

**Soft Delete**
- [ ] `softDelete()` service method sets `deletedAt: new Date()` — no `prisma.delete()` calls
- [ ] `findAll()` and `findOne()` filter `deletedAt: null` by default
- [ ] Deletion logs the `deletedByUserId`

**Transactions**
- [ ] Multi-write operations use `prisma.$transaction(async (tx) => { ... })`
- [ ] All writes inside the callback use `tx`, not `this.prisma`
